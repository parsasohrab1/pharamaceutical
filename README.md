در ادامه، یک سند **SRS (Software Requirements Specification)** کامل برای محصول «سامانه شبیه‌ساز ترکیبی کوآنتوم-کلاسیک برای طراحی و بهینه‌سازی مولکول‌های دارویی» تهیه شده است. این سند بر اساس استاندارد IEEE 830 و با توجه به نیازهای ثبت اختراع و تجاری‌سازی در ایران تدوین شده است.

**نسخه:** 1.1  
**تاریخ:** ۱۴۰۵/۰۳/۲۰

## ۱. مقدمه

### ۱.۱ هدف
این سند الزامات عملکردی و غیرعملکردی سامانه **HQCA** را تعریف می‌کند؛ سامانه‌ای که با ترکیب محاسبات کوآنتومی و یادگیری ماشین کلاسیک، پیش‌بینی میزان اتصال مولکول‌های دارویی به پروتئین‌های هدف را با دقت بالا و هزینه محاسباتی بهینه انجام می‌دهد.

### ۱.۲ دامنه
سامانه قادر است:
- دریافت رشته SMILES مولکول دارو و توالی پروتئین هدف
- تولید داده‌های سنتتیک در صورت کمبود داده واقعی
- شبیه‌سازی انرژی آزاد اتصال با استفاده از الگوریتم VQE روی شبیه‌ساز کوآنتومی یا سخت‌افزار واقعی
- خروجی: نمره اتصال (Binding Affinity) در محدوده ۰ تا ۱۰۰ و احتمال موفقیت

### ۱.۳ تعاریف و اختصارات
| عبارت | تعریف |
|-------|-------|
| QML | یادگیری ماشین کوانتومی |
| VQE | Variational Quantum Eigensolver |
| HQCA | سامانه ترکیبی پیشنهادی |
| SMILES | نمایش رشته‌ای ساختار مولکولی |
| PDB | پایگاه داده ساختار پروتئین |
| Angle Embedding | نگاشت داده به زوایای چرخش کیوبیت |

### ۱.۴ مراجع
- استاندارد IEEE 830-1998
- قوانین ثبت اختراع ایران (مصوب ۱۳۸۶ و اصلاحات)
- مقالات مرجع: PocketGen (2025)، ChemBFN (2025)، Q-BAFNet (2025)

## ۲. شرح کلی

### ۲.۱ چشم‌انداز محصول
HQCA یک سامانه نرم‌افزاری به صورت SaaS (نرم‌افزار به عنوان خدمت) است که به شرکت‌های داروسازی، دانشگاه‌ها و مراکز تحقیقاتی امکان می‌دهد فرآیند غربالگری مجازی مولکول‌ها را با سرعت ۲۰ برابر روش‌های کلاسیک و دقت بالای ۸۵٪ انجام دهند.

### ۲.۲ ویژگی‌های اصلی
| شماره | ویژگی |
|-------|-------|
| F1 | تولید داده‌های سنتتیک جیب پروتئین و مولکول دارو |
| F2 | محاسبه توصیفگرهای کلاسیک (MW، LogP، HBA، HBD و ...) |
| F3 | Encoding کوانتومی با Angle Embedding |
| F4 | مدار کوانتومی وردشی (VQC) با الگوی درهم‌تنیدگی درخت دودویی |
| F5 | بهینه‌ساز کلاسیک COBYLA |
| F6 | پیش‌بینی نمره اتصال و نمایش گرافیکی جیب اتصال |
| F7 | ذخیره و بازیابی نتایج (پایگاه داده) |
| F8 | گزارش‌گیری جهت ثبت اختراع و مستندات آزمایشگاهی |

### ۲.۳ کاربران
| نقش | توضیح |
|------|-------|
| محقق داروسازی | وارد کردن مولکول و پروتئین، دریافت پیش‌بینی |
| مدیر سامانه | نصب، پیکربندی، نظارت بر پردازش‌های کوانتومی |
| توسعه‌دهنده QML | تنظیم پارامترهای مدار وردشی و بهینه‌ساز |

### ۲.۴ محدودیت‌ها
- نبود سخت‌افزار کوآنتومی فیزیکی در ایران → در فاز اول، شبیه‌سازهای کلاسیک (Qiskit Aer، PennyLane) جایگزین می‌شوند.
- حداکثر ۳۰ کیوبیت شبیه‌سازی در ماشین‌های با ۶۴ گیگابایت رم.
- زمان پاسخگویی برای هر جفت دارو-پروتئین حداکثر ۵ دقیقه (در حالت شبیه‌ساز).

### ۲.۵ تفکیک نسخه MVP و نسخه کامل محصول
برای جلوگیری از ابهام در دامنه تحویل، محصول به دو سطح تعریف می‌شود:

| سطح محصول | دامنه |
|-----------|-------|
| MVP قابل استفاده | دریافت SMILES/FASTA، اعتبارسنجی RDKit/FASTA، اجرای پیش‌بینی با backend شبیه‌سازی یا مدل آموزش‌دیده، تولید نمره اتصال، confidence، مختصات جیب، گزارش CSV/PDF/PDB، تاریخچه درخواست، احراز هویت پایه، ذخیره نتایج و تست‌های یکپارچه FR-01 تا FR-19. |
| نسخه کامل تجاری/علمی | اتصال به دیتاست واقعی بزرگ BindingDB/PDBbind، آموزش و اعتبارسنجی رسمی مدل، backendهای کوانتومی قابل انتخاب، PocketGen/ChemBFN واقعی یا جایگزین اعتبارسنجی‌شده، مشاهده سه‌بعدی پیشرفته ساختار PDB، صف پردازش production، استقرار سازمانی، مانیتورینگ، RBAC کامل و معیارهای علمی قابل دفاع در گزارش محصول. |

MVP نباید به‌عنوان اثبات نهایی دقت علمی یا جایگزین آزمایشگاهی معرفی شود؛ هدف MVP اثبات مسیر end-to-end محصول، قراردادهای API، تجربه کاربری و قابلیت آموزش/ارزیابی اولیه است.

### ۲.۶ سناریوهای کاربری اصلی

#### ۲.۶.۱ محقق داروسازی
1. ورود به سامانه و انتخاب/ایجاد پروژه تحقیقاتی.
2. وارد کردن SMILES مولکول و توالی FASTA پروتئین هدف.
3. اجرای پیش‌بینی و مشاهده `binding_score`، انرژی اتصال، confidence و مرکز جیب اتصال.
4. بررسی نمودار نمره اتصال و نمایش سه‌بعدی جیب.
5. دانلود گزارش CSV/PDF/PDB و مقایسه نتایج با درخواست‌های قبلی.

#### ۲.۶.۲ مدیر سامانه
1. پیکربندی اتصال PostgreSQL، MinIO، Redis/RQ و کلیدهای امنیتی.
2. مدیریت کاربران، نقش‌ها، لاگ‌ها و سلامت سرویس.
3. پایش jobهای صف پردازش و کنترل خطاهای API/worker.
4. اطمینان از رمزنگاری داده‌های حساس و پشتیبان‌گیری از داده‌ها و artifactها.

#### ۲.۶.۳ توسعه‌دهنده QML
1. آماده‌سازی دیتاست BindingDB/PDBbind و اجرای pipeline آموزش.
2. مقایسه baseline کلاسیک با پارامترهای VQC/شبیه‌ساز کوانتومی.
3. ذخیره و نسخه‌بندی artifact مدل و پارامترهای VQC.
4. اجرای benchmark علمی و تحلیل MAE، confidence calibration و drift داده.

### ۳.۸ فرمول استاندارد تبدیل انرژی اتصال به Binding Score
برای همه خروجی‌های محصول، تبدیل انرژی اتصال به نمره ۰ تا ۱۰۰ باید با فرمول زیر انجام شود:

```text
score = 100 * (E_max - E) / (E_max - E_min)
E_min = -15.0 kcal/mol
E_max = -0.1 kcal/mol
```

که در آن `E` همان `binding_energy_kcal_mol` است. خروجی فرمول به بازه `[0, 100]` محدود می‌شود. انرژی اتصال منفی‌تر نشان‌دهنده اتصال قوی‌تر است، بنابراین مقدار `-15.0` به نمره ۱۰۰ و مقدار `-0.1` به نمره ۰ نگاشت می‌شود. اگر دامنه کالیبراسیون با داده واقعی تغییر کند، مقادیر `E_min` و `E_max` باید در artifact مدل، مستندات API و گزارش خروجی ثبت شوند.

### ۳.۹ روش محاسبه Confidence
در MVP، confidence یک برآورد عملیاتی از کیفیت ورودی و وضعیت مدل است، نه احتمال قطعی موفقیت آزمایشگاهی. فرمول MVP:

```text
descriptor_coverage = نسبت توصیفگرهای نرمال‌شده که در مرز ۰ یا ۱ clip نشده‌اند
sequence_factor = min(length(FASTA), 60) / 60
confidence = clip(55 + 30 * descriptor_coverage + 10 * sequence_factor, 50, 95)
```

در نسخه کامل، confidence باید با روش آماری کالیبره شود، مانند ensemble variance، cross-validation calibration، conformal prediction یا bootstrap روی مدل آموزش‌دیده. معیار پذیرش confidence در نسخه کامل این است که بازه‌های اطمینان گزارش‌شده روی مجموعه آزمون مستقل کالیبره باشند؛ برای مثال، حدود ۹۰٪ نمونه‌ها باید در بازه اطمینان ۹۰٪ قرار گیرند.

| همبستگی رتبه‌ای | Spearman correlation بین انرژی پیش‌بینی‌شده و انرژی واقعی روی مجموعه آزمون. هدف پیشنهادی: حداقل ۰.۶۵. |
| طبقه‌بندی اتصال قوی/ضعیف | تبدیل انرژی واقعی و پیش‌بینی‌شده به کلاس قوی/متوسط/ضعیف با آستانه‌های از پیش تعریف‌شده و گزارش accuracy/F1. ادعای «۸۵٪» فقط باید به همین accuracy/F1 و همین آستانه‌ها ارجاع دهد. |
| کالیبراسیون confidence | بررسی اینکه بازه‌های confidence با نرخ پوشش واقعی سازگار باشند؛ مثلاً بازه ۹۰٪ باید نزدیک به ۹۰٪ پوشش واقعی داشته باشد. |
| مقایسه baseline | مقایسه VQC با حداقل یک baseline کلاسیک مانند RandomForest یا XGBoost روی split یکسان. |

هر گزارش علمی باید شامل نسخه دیتاست، روش split، seed، تعداد نمونه train/test، روش حذف leakage، نسخه مدل، نسخه descriptorها و artifactهای آموزش باشد.

## ۹. پیوست‌ها (پیشنهادی)

### پیوست A – نمونه شبه‌کد ماژول VQC
```python
from qiskit import QuantumCircuit
import numpy as np

def angle_embedding(features, qc, qubits):
    for i, f in enumerate(features):
        qc.rx(np.arctan(f), qubits[i])

def tree_entanglement(qc, n_qubits):
    for layer in range(int(np.log2(n_qubits))):
        step = 2**layer
        for i in range(0, n_qubits, step*2):
            for j in range(step):
                if i+j+step < n_qubits:
                    qc.cx(i+j, i+j+step)

def variational_circuit(features, params):
    n = len(features)
    qc = QuantumCircuit(n)
    angle_embedding(features, qc, range(n))
    for p in params:
        for i in range(n):
            qc.ry(p[i], i)
        tree_entanglement(qc, n)
    return qc
```
